## 📋 项目核心信息

### 项目概述
- **名称**: 情绪驱动机器人
- **功能**: 通过 AI 识别用户情绪/故事，生成机器人动作序列
- **架构**: WebApp → OpenClaw → AI 分析 → Webhook → Milk DuoS 硬件

### 技术栈
| 层级 | 组件 | 技术 |
|------|------|------|
| 前端 | WebApp | React/Vue/HTML |
| 中台 | OpenClaw | Node.js + Skills |
| AI | 情绪识别 | qwen3.5-omni-plus-realtime |
| 推送 | Webhook | 自研 HTTP 服务 |
| 硬件 | 执行端 | Milk DuoS 开发板 |

---

## ✅ 已完成功能

### 1. 基础情绪识别（7 种）
- happy（开心）、sad（难过）、scared（害怕）、grateful（感谢）、calm（平静）、angry（生气）、surprised（惊讶）
- 准确率：98%+
- 延迟：~2 秒

### 2. 动作库（11 个）
| 编号 | 动作 | 说明 |
|------|------|------|
| 0 | 站立 | 待机 |
| 1 | 挥手 | 开心 |
| 2 | 摇头 | 否定 |
| 3 | 一边致谢 | 感谢左 |
| 4 | 另一边致谢 | 感谢右 |
| 5 | 交通指挥 | 特殊 |
| 6 | 舞蹈歌曲 | 庆祝 |
| 7 | 自由飞翔 | 特殊（原金鸡独立） |
| 8 | 下蹲 | 害怕 |
| 9 | 前进 2 步 | 接近 |
| 10 | 后退 2 步 | 后退 |

### 3. 情绪→动作映射
- 支持强度分级（轻度/中度/重度）
- 支持故事型输入
- 映射表：`skills/robot-behavior/references/emotion-map.md`

### 4. Webhook 队列服务
**文件**: `services/webhook-receiver/server.py`

**端点**:
- `POST /emotion` - 接收情绪数据，AI 分析后入队
- `POST /action/{client_id}` - 直接发送动作到队列
- `GET /poll/{client_id}` - 硬件轮询获取动作
- `POST /ack/{client_id}` - 确认动作完成
- `GET /health` - 健康检查
- `GET /cleanup` - 手动清理队列

**特性**:
- ✅ 队列定期清理（每 3 小时，保留 3 小时内命令）
- ✅ 最新优先策略（新输入时清除未 ACK 旧命令）
- ✅ 轮询标记为 processing（避免重复返回）

### 5. 导演模式（概念设计）
**文件**: 
- `skills/robot-behavior/scripts/director_video.py`（测试脚本）
- `skills/robot-behavior/references/director-prompt.md`（提示词设计）
- `skills/robot-behavior/references/action-variants.md`（动作变体映射）

**设计思路**:
1. 并发提取情绪标签（~3 秒）
2. 导演模型分析情绪曲线（~1.5 秒）
3. 根据风格选择动作变体（避免重复）
4. 发送到队列

**状态**: ⚠️ **概念验证完成，未集成到生产环境**

## 📊 测试结果

### 长故事测试（5 段落）
**故事**: 小明的一天

| 方案 | 延迟 | 重复数 | 高潮 | 推荐度 |
|------|------|--------|------|--------|
| 完整输入 | ~2 秒 | - | ❌ | ❌ |
| 分段输入 | ~3 秒 | 2 处 | ❌ | ⚠️ |
| 导演模式 | ~4.5 秒 | 0 处 | ✅ | ✅ |

### 导演模式输出（测试）
```
段落 1: happy + formal → [1, 0] 挥手→站立
段落 2: grateful + formal → [3, 4, 0] 致谢→致谢→站立
段落 3: scared + climax → [10, 8, 10, 0] 后退→下蹲→后退→站立 ⭐
段落 4: grateful + warm → [3, 1, 0] 致谢→挥手→站立 ✅
段落 5: happy + warm → [1, 6, 0] 挥手→舞蹈→站立
```
