## 一、需求背景

### 当前限制

现有 `robot-behavior` 技能单次调用仅能生成 3-5 个动作的序列，适合短情绪表达（如"我很难过"→[2,8,0]）。但面对长故事演绎场景（如儿童故事、情景剧），存在以下限制：

1. **长度限制**：故事超过 100 字时，AI 响应可能被截断
2. **单情绪限制**：单次分析仅识别一种主导情绪，无法处理多情节变化
3. **演绎不连贯**：无法表现故事中情绪的起伏变化

### 用户场景

```
输入："早上小明起床后开心地挥手跟妈妈打招呼。
      吃完早餐他背着书包出门，路上遇到邻居爷爷，他鞠躬致谢问好。
      到了学校发现忘记带作业，他吓得蹲在地上。
      老师走过来安慰他，他感动地挥手。
      放学后回家，看到妈妈准备了生日蛋糕，他兴奋地跳起了舞蹈。"

期望：机器人按情节顺序演绎 5 段不同情绪的动作
  - 段落 1: 开心 → 挥手→舞蹈
  - 段落 2: 感谢 → 致谢→致谢
  - 段落 3: 害怕 → 下蹲→后退
  - 段落 4: 感动 → 挥手
  - 段落 5: 兴奋 → 舞蹈
```

## 二、技术方案

### 2.1 整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│ 1. 故事预处理层                                                   │
│    - 文本分段（按句号/感叹号/问号）                                │
│    - 段落过滤（去除过短片段）                                     │
│    - 段落合并（相邻相似情绪可合并）                                │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 2. 并行情绪分析层                                                 │
│    - 多段落并发调用 AI 模型（提升速度）                              │
│    - 单段落情绪识别 + 动作生成                                     │
│    - 结果验证与重试机制                                          │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 3. 队列管理层                                                    │
│    - 按顺序添加多个命令到队列                                     │
│    - 每个命令包含 segment_index 字段                              │
│    - 支持批量入队和事务保证                                       │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 4. 硬件执行层                                                    │
│    - 批量轮询获取多个命令                                         │
│    - 按 segment_index 顺序执行                                    │
│    - 段落间停顿（增强演绎效果）                                   │
│    - 逐段 ACK 确认                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 核心流程

```
Step 1: 故事拆分
────────────────
输入：长故事文本
输出：段落列表 ["段落 1", "段落 2", ...]

拆分规则：
- 按 [。！？.!?] 拆分
- 过滤长度 < 5 的段落
- 如果段落数=1，尝试按 [，,] 二次拆分


Step 2: 分段情绪分析
────────────────────
对每个段落调用 qwen3.5-omni-plus-realtime

输入：段落文本
输出：{status, action_sequence, emotion_detected, rationale}

优化：
- 并发调用（5 个段落可同时进行）
- 超时时间缩短为 10 秒/段
- 失败段落自动重试 1 次


Step 3: 批量入队
────────────────
依次发送每个段落的动作序列到 Webhook 队列

POST /action/{client_id}
{
  "action_sequence": [1, 6, 0],
  "emotion": "happy",
  "segment_index": 1,      // 新增：段落序号
  "total_segments": 5,     // 新增：总段落数
  "command_id": "cmd_xxx"
}


Step 4: 批量轮询
────────────────
硬件端一次性获取所有待执行命令

GET /poll/{client_id}?batch=true&max=10

响应：
{
  "status": "success",
  "batch": true,
  "count": 5,
  "commands": [
    {"command_id": "cmd_1", "action_sequence": [...], "segment_index": 1},
    {"command_id": "cmd_2", "action_sequence": [...], "segment_index": 2},
    ...
  ]
}


Step 5: 顺序执行
────────────────
硬件端按 segment_index 顺序执行每个命令

for command in commands:
    execute(command.action_sequence)
    wait(动作执行时间)
    if command.segment_index < total_segments:
        wait(段落停顿时间，如 1 秒)  // 增强演绎效果
    send_ack(command.command_id)
```

## 四、实现细节

### 4.1 故事拆分算法

```python
import re

def split_story(story: str) -> list:
    # 一级拆分：按句子结束符
    sentences = re.split(r'[。！？.!?]', story)
    segments = [s.strip() for s in sentences if len(s.strip()) > 5]
    
    # 如果段落太少，尝试二级拆分：按逗号
    if len(segments) <= 1:
        segments = [s.strip() for s in re.split(r'[，,]', story) if len(s.strip()) > 5]
    
    return segments
```

### 4.2 并发分析优化

```python
import concurrent.futures

def analyze_segments_parallel(segments: list, api_key: str, max_workers: int = 5) -> list:
    results = []
    
    with concurrent.futures.ThreadPoolExecutor(max_workers=max_workers) as executor:
        futures = {executor.submit(analyze_segment, seg, api_key): i for i, seg in enumerate(segments)}
        
        for future in concurrent.futures.as_completed(futures):
            index = futures[future]
            result = future.result()
            results.append((index, result))
    
    # 按原始顺序排序
    results.sort(key=lambda x: x[0])
    return [r[1] for r in results]
```

### 4.3 JSON 截断容错

```python
def parse_truncated_json(full_text: str) -> dict:
    """处理被截断的 JSON 响应"""
    try:
        return json.loads(full_text)
    except json.JSONDecodeError:
        # 尝试提取 action_sequence
        seq_match = re.search(r'"action_sequence":\s*\[([\d,\s]+)\]', full_text)
        if seq_match:
            sequence = [int(x.strip()) for x in seq_match.group(1).split(',')]
            
            emo_match = re.search(r'"emotion_detected":"([^"]+)"', full_text)
            emotion = emo_match.group(1) if emo_match else "unknown"
            
            return {
                "status": "success",
                "action_sequence": sequence,
                "emotion_detected": emotion,
                "rationale": ""
            }
        
        return {"status": "error", "error_message": "无法解析响应"}
```

## 七、风险与应对

### 风险 1：API 调用失败

**场景**：某段落分析超时或失败

**应对**：
- 单段失败不影响其他段落
- 失败段落标记为 "unknown"，使用默认动作 [0]
- 日志记录失败详情

### 风险 2：队列溢出

**场景**：超长故事超过 100 条命令

**应对**：
- 预处理阶段限制最大段落数（如 50）
- 超出部分截断并提示用户
- 或分批发送（高级功能）

### 风险 3：执行顺序错乱

**场景**：硬件端未按 segment_index 顺序执行

**应对**：
- 命令中包含 segment_index 和 total_segments
- 硬件端按序号排序后执行
- ACK 确认时检查序号连续性

### 风险 4：段落停顿时间不当

**场景**：停顿太短（演绎不清晰）或太长（用户等待）

**应对**：
- 默认 1.5 秒，可配置
- 根据动作序列长度动态调整
- 支持用户在故事中插入停顿标记（如"..."）

## 十、总结

长故事演绎功能通过**分段处理 + 批量执行**的策略，解决了单次分析长度限制和单情绪限制的问题。方案保持与现有系统的兼容性，通过新增脚本和扩展 API 实现，不影响现有短情绪表达功能。

**核心优势**：
1. 支持任意长度故事演绎
2. 保持情节连贯性和情绪变化
3. 兼容现有单段分析模式
4. 可扩展为儿童故事机、情景剧表演等应用场景

**建议实现顺序**：先实现核心功能验证可行性，再优化性能和用户体验。
