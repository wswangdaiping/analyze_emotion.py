## 1. Webhook 接口规范

### 1.1 接收端点 (OpenClaw → 本地服务)

| 端点 | 方法 | 说明 |
|------|------|------|
| `/emotion` | POST | 接收情绪数据并分析 |
| `/action/{client_id}` | POST | 直接推送动作序列 |
| `/poll/{client_id}` | GET | 客户端轮询获取命令 |
| `/ack/{client_id}` | POST | 客户端确认命令执行 |
| `/health` | GET | 健康检查 |
| `/cleanup` | GET | 手动清理过期命令 |

### 1.2 POST /emotion 请求格式

**请求头:**
```
Content-Type: application/json
```

**请求体:**
```json
{
  "input_type": "text",           // 必填：输入类型
  "content": "我今天很开心",        // 必填：情绪文本内容
  "session_id": "user_001"        // 可选：会话 ID
}
```

**响应体 (成功):**
```json
{
  "status": "success",
  "emotion": "happy",
  "action_sequence": [1, 6, 0],
  "command_id": "cmd_abc123",
  "client_id": "milk_duos_001"
}
```

**响应体 (错误):**
```json
{
  "status": "error",
  "error_code": "INVALID_INPUT",
  "error_message": "Missing 'content' field"
}
```

### 1.3 POST /action/{client_id} 请求格式

**请求体:**
```json
{
  "action_sequence": [1, 6, 0],   // 必填：动作编号数组
  "emotion": "happy"               // 可选：情绪类型
}
```

**响应体:**
```json
{
  "status": "success",
  "command_id": "cmd_xyz789"
}
```

### 1.4 GET /poll/{client_id} 响应格式

**有待执行命令:**
```json
{
  "command_id": "cmd_xyz789",
  "action_sequence": [1, 6, 0],
  "emotion": "happy",
  "created_at": "2026-05-03T12:00:00Z",
  "status": "pending"
}
```

**无命令:**
```json
{
  "status": "no_action",
  "message": "No pending actions"
}
```

### 1.5 POST /ack/{client_id} 请求格式

**请求体:**
```json
{
  "command_id": "cmd_xyz789"      // 必填：要确认的命令 ID
}
```

## 2. 动作序列格式

### 2.1 数据结构

```json
{
  "action_sequence": [number],    // 动作编号数组，长度 1-5
  "action_names": ["string"],     // 动作名称数组 (可选)
  "emotion_detected": "string",   // 识别的情绪类型
  "rationale": "string"           // 生成理由说明
}
```

### 2.2 动作编号定义

| 编号 | 动作名称 | 说明 |
|------|---------|------|
| 0 | 站立 | 中立姿态、待机 |
| 1 | 挥手 | 打招呼、开心 |
| 2 | 摇头 | 否定、拒绝 |
| 3 | 一边致谢 | 感谢 (左) |
| 4 | 另一边致谢 | 感谢 (右) |
| 5 | 交通指挥 | 特殊演绎 |
| 6 | 舞蹈歌曲 | 庆祝、非常开心 |
| 7 | 自由飞翔 | 特殊演绎 |
| 8 | 下蹲 | 害怕、躲藏、低落 |
| 9 | 前进 2 步 | 主动接近 |
| 10 | 后退 2 步 | 害怕后退 |

### 2.3 动作序列规则

- ✅ **必须以 0 结尾** (回归待机状态)
- ✅ **推荐长度 1-5 个动作**
- ✅ **每个动作间隔 500ms 执行**
- ⚠️ **空序列视为无效**，默认返回 `[0]`

**示例:**
```json
// 开心 (中度)
{
  "action_sequence": [1, 6, 0],
  "action_names": ["挥手", "舞蹈歌曲", "站立"],
  "emotion_detected": "happy",
  "rationale": "检测到'开心'情绪，选择挥手和舞蹈动作表达喜悦"
}
```

## 3. 技能调用约定

### 3.1 robot-behavior Skill

**功能**: 情绪文本 → 动作序列

**调用方式:**
```bash
python3 scripts/analyze_emotion.py --input "我很难过" --pretty
```

**输入:**
```json
{
  "input_type": "text",
  "content": "用户输入的文本内容",
  "session_id": "可选"
}
```

**输出:**
```json
{
  "status": "success",
  "action_sequence": [2, 8, 0],
  "action_names": ["摇头", "下蹲", "站立"],
  "emotion_detected": "sad",
  "rationale": "检测到'难过'情绪，选择摇头和下蹲表达低落"
}
```

**情绪类型:** `happy`, `sad`, `nervous`, `scared`, `angry`, `grateful`, `calm`, `surprised`

### 3.2 json-webhook-skill

**功能**: 发送 JSON 数据到 Webhook 地址

**调用方式:**
```bash
python scripts/send_webhook.py \
  --url http://localhost:8765/action/milk_duos_001 \
  --data '{"action_sequence": [1, 6, 0]}'
```

**参数:**
- `--url`: 目标 Webhook URL
- `--data`: JSON 字符串
- `--header`: 可选，认证头 (如 `Authorization=Bearer xxx`)
- `--timeout`: 可选，超时秒数 (默认 30)

## 5. 轮询接口规范

### 5.1 轮询频率

- **推荐间隔**: 2 秒
- **最低间隔**: 1 秒 (避免限流)
- **最高间隔**: 10 秒 (可能增加延迟)

### 5.2 轮询流程

```
1. DuoS 发送 GET /poll/{client_id}
2. 服务器返回待执行命令 (或空响应)
3. 如有命令，状态自动标记为 "processing"
4. DuoS 执行动作序列
5. DuoS 发送 POST /ack/{client_id} 确认完成
6. 服务器标记命令为 "completed"
```

### 5.3 客户端 ID 规范

- **格式**: `milk_duos_001` (小写字母 + 数字 + 下划线)
- **注册**: 首次轮询自动注册
- **白名单**: 生产环境建议启用白名单

### 5.4 认证方式

**当前**: 无认证 (内网环境)

**生产环境建议:**
```bash
# 方式 A: Bearer Token
Authorization: Bearer your_token_here

# 方式 B: API Key
X-API-Key: your_api_key

# 方式 C: 客户端证书
(HTTPS 双向认证)
```

## 6. 队列管理策略

### 6.1 队列配置

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `CLEANUP_INTERVAL_HOURS` | 3 | 清理间隔 (小时) |
| `MAX_AGE_HOURS` | 3 | 命令最大保留时间 |
| `KEEP_LATEST_ONLY` | true | 新输入时清除未 ACK 旧命令 |

### 6.2 清理策略

```
当前时间：2026-05-03 19:00:00
保留时间：3 小时
清理截止时间：2026-05-03 16:00:00

结果:
- 16:00:00 之后的命令 → 保留 ✅
- 16:00:00 之前的命令 → 清理 ❌
```

### 6.3 最新优先策略

当 `KEEP_LATEST_ONLY = true` 时:
- 新情绪输入到达时，自动清除该客户端所有未 ACK 的旧命令
- 确保机器人始终执行最新的情绪表达
- 避免命令堆积导致延迟

## 10. 故障排查

### 10.1 常见问题

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| 端口被占用 | 8765 已被使用 | `lsof -i :8765` 查找并关闭 |
| 情绪分析失败 | API Key 无效 | 检查 `DASHSCOPE_API_KEY` |
| 轮询无响应 | 队列为空 | 正常，继续轮询 |
| 命令不执行 | 未收到 ACK | 检查 ACK 端点调用 |

### 10.2 日志位置

# 默认日志目录
