## 山东大学项目实训-个人博客（二）：项目第二阶段——迁移到 OpenClaw 平台并完成第一版 Skill 落地

@[TOC](文章目录)

### 1. 背景：为什么迁移到 OpenClaw

项目早期自研 FastAPI + WebSocket + ChatPipeline 架构主要用于验证"文本输入 → 语义理解 → 行为生成 → 机器人执行"的完整链路。后续迁移到 OpenClaw Skill 后，系统不再把重点放在自建 Agent 通信和调度框架上，而是将核心能力收敛到 Skill 内部的故事解析、动作序列生成和机器人行为表达。

OpenClaw 是一个面向多渠道交互的本地优先 Agent 平台，负责承载会话、渠道、工具调用和 Agent 运行环境。在本项目中，机器人行为生成逻辑被封装为 OpenClaw Skill：通过 `SKILL.md` 描述该能力的用途、触发场景和调用方式，使 Agent 能够在合适的上下文中执行故事解析、动作序列生成与结果返回。

### 2. 本周工作总结

本周完成了 robot-behavior skill（机器人行为生成技能）的核心开发工作，实现了从用户情绪/故事输入到机器人动作序列的完整转换链路。具体包括：

- **情绪响应模块**：识别用户输入文本中的情绪（开心、难过、紧张、感谢等），并映射为相应的机器人动作序列
- **故事演绎模块**：解析用户讲述的简短故事情节，提取角色情绪变化和动作描述，生成轻量级表演动作序列
- **输入输出格式定义**：设计并实现了标准化的 JSON 输入输出格式，支持 `session_id` 跟踪和完整的错误处理机制
- **动作映射表**：定义了 10 种基础动作（站立、挥手、摇头、致谢、舞蹈、下蹲、前进、后退等），并建立了情绪到动作的映射关系

### 3. 技术实现与挑战

- **核心功能实现**：
  - 设计了基于大语言模型的情绪理解流程，输入文本经 API 分析后输出结构化 JSON
  - 实现了动作序列生成器，确保生成的动作数量在 2-5 个之间，符合机器人轻量级表演的需求
  - 建立了情绪分类体系：`happy`、`sad`、`nervous`、`grateful`、`neutral` 等，并映射到对应的动作编号

- **技术难题与解决方案**：
  - **动作序列合理性**：初期生成的动作序列存在逻辑跳跃问题，例如从"开心"直接跳到"下蹲"。通过引入情绪过渡规则和故事情绪跟随原则（按照情绪变化顺序生成动作），使动作序列更加自然流畅
  - **输入歧义处理**：用户输入可能同时包含多种情绪或隐含意图。通过定义明确的澄清策略和错误码（`INVALID_INPUT` / `EMOTION_UNKNOWN` / `STORY_INVALID`），确保系统在无法识别时能够优雅降级

- **技术选型的思考**：
  - 选择 JSON 作为输入输出格式，便于与 duo 端进行数据交换
  - 动作使用数字编码而非字符串，减少数据传输量，提高机器人解析效率

### 4. 使用AI生成代码：提示词/VIBE/SPEC过程

在第一版 Skill 开发中，我主要把 AI 用在两个环节：一是根据明确的输入输出约束生成情绪映射和动作序列逻辑，二是用真实故事输入反复验证动作序列是否符合机器人表达需求。VIBE 和 SPEC 没有作为单独的方法论展开，而是落实在具体开发流程中：先明确 Skill 的输入输出、动作编号、JSON 格式和降级规则，再用样例输入检查结果是否稳定。

**实际使用方式 1 - 情绪映射函数生成：**

```
请生成一个 Python 函数，输入为用户文本字符串，输出为情绪标签和对应的机器人动作序列。
情绪包括：happy, sad, nervous, grateful, neutral。
动作映射规则：开心→[1, 6]（挥手/舞蹈），难过→[2, 8]（摇头/下蹲），
紧张→[8, 10]（下蹲/后退），感谢→[3, 4]（致谢），中立→[0]（站立）。
返回格式：{"emotion": "happy", "actions": [1, 6], "rationale": "用户表达开心情绪"}
```

**实际使用方式 2 - 故事解析逻辑：**

```
请生成一个函数，输入为简短故事文本，输出为按情绪变化顺序排列的动作序列。
要求：
1. 识别故事中的关键情绪转折点
2. 每个情绪阶段映射为 1-2 个动作
3. 总动作数不超过 5 个
4. 动作编号参考映射表：0-站立, 1-挥手, 2-摇头, 8-下蹲, 9-前进, 10-后退
```

通过这种方式，第一版 Skill 很快形成了稳定的基础能力：能识别简单情绪，能生成符合 JSON 格式的动作序列，也能通过动作编号与机器人端执行逻辑对接。后续主要问题不在于接口格式，而在于复杂故事表达时，单一情绪标签会压缩掉叙事过程中的时序变化。

### 5. 技术理解

- 通过这个项目，深入理解了情绪识别在人机交互中的应用场景。情绪到动作的映射不仅仅是简单的键值对应，更需要考虑情绪过渡的自然性和上下文的连贯性
- 学习了如何使用 AI 辅助开发，从需求描述到代码生成的迭代过程。关键在于：精确描述输入输出格式、提供充分的示例、建立验证机制确保生成质量
- 对机器人行为设计的理解更加深入——轻量级表演（2-5 个动作）比复杂编舞更实用，用户更容易理解机器人的意图

### 6. 阶段小结与遗留问题

第一版 robot-behavior Skill 完成了基础情绪映射和动作序列生成，系统可以从文本输入生成结构化 JSON，并通过动作编号表达机器人的基础行为。这个阶段的重点是把能力从自研 Agent 服务迁移到 OpenClaw Skill 中，验证 Skill 描述、触发、调用和结果返回的完整流程。

这一版也暴露出一个核心问题：情绪标签是一种有损编码。对于"我很开心"这类单句输入，`happy → [1, 6]` 的映射是有效的；但面对复杂故事时，故事中的入场、转折、受挫、发现线索、离开等过程会被压缩成单一情绪标签，最终只能输出一组相对静态的动作序列，难以体现叙事的时序结构。

因此，下一阶段需要从"情绪标签驱动动作"进一步演进到"故事结构驱动动作"：先解析故事情节节点，再根据每个片段生成动作序列，最后合并为完整的机器人行为表达。这个问题会在下一阶段通过故事解析方案继续解决。
