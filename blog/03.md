# 从 Skill 输出到机器人执行：Webhook 队列的工程落地

上一阶段完成了 `robot-behavior` Skill 的基础能力，本阶段的重点是解决"Skill 输出的动作序列如何可靠地传递给机器人端执行"这个工程问题。也就是说，系统不只要能生成动作，还要能让 Duo 板稳定拿到动作、执行动作，并把执行结果确认回来。

---

## 1. 技术实现

### 1.1 轮询架构设计

这一阶段采用**轮询模式**：Duo 板主动向 Webhook Server 拉取待执行命令，而不是由云端主动推送。

```
WebApp
  │
  │ POST /emotion
  ▼
Webhook Receiver
  │
  │ 调用 robot-behavior Skill 分析情绪 / 生成动作序列
  ▼
动作队列（每个 client_id 独立）
  │
  │ GET /poll/{client_id}  每 2 秒轮询
  ▼
Duo 板执行动作
  │
  │ POST /ack/{client_id}
  ▼
Webhook Receiver 标记命令完成
```

**为什么选择轮询而不是推送：**

- Duo 板可能处在内网环境，云端主动推送需要处理公网 IP、端口暴露或 NAT 穿透问题
- 轮询模型部署简单，机器人端只需要定时访问固定 HTTP 端点
- 对本项目的动作执行场景来说，0-2 秒的轮询等待可以接受
- Webhook Server 可以通过队列状态管理和 ACK 机制保证命令不会丢失

这一设计牺牲了一部分实时性，但换来了更稳定的部署方式和更低的联调复杂度。

---

### 1.2 端点设计

Webhook Receiver 负责接收 Skill 输出、维护动作队列，并提供给机器人端轮询。

| 端点 | 方法 | 作用 |
|------|------|------|
| `/emotion` | POST | 接收用户文本，调用 Skill 生成动作序列并入队 |
| `/action/{client_id}` | POST | 直接向指定客户端写入动作序列，便于测试 |
| `/poll/{client_id}` | GET | Duo 板轮询待执行命令 |
| `/ack/{client_id}` | POST | Duo 板确认命令执行完成 |
| `/health` | GET | 服务健康检查 |

核心响应格式保持简单：有命令时返回 `command_id`、`action_sequence`、`emotion`、`created_at` 和 `status`；无命令时返回 `no_action`。这样 Duo 板只需要处理两类结果：执行命令，或者继续等待。

---

### 1.2.5 早期版本的问题

早期版本的 Webhook Receiver 没有队列状态管理，动作序列写入后直接返回给轮询端，没有追踪机制。这导致两个实际问题：第一，Duo 板轮询到动作后如果执行失败，系统无从感知，命令就此丢失；第二，多个请求同时写入时，队列内容会被覆盖，后写入的动作序列会替换掉还未被取走的旧命令。这两个问题直接推动了 command_id 追踪机制和 threading.Lock 并发保护的引入。

---

### 1.3 核心模块

**robot-behavior Skill**：负责理解输入文本并生成动作序列。情绪到动作的基础映射规则已在第二阶段定义，这里只保留为 Skill 的内部能力，不再重复展开。
**Webhook Receiver**：HTTP 服务管理动作队列，核心功能包括：

- 每个客户端维护独立队列，使用 `client_id` 区分不同机器人
- 每条命令生成唯一 `command_id`，用于执行追踪和 ACK 确认
- 队列项维护状态：`pending`、`processing`、`completed`
- 支持手动动作写入，方便绕过 Skill 做机器人端联调
- 支持健康检查，便于部署后确认服务是否可用
---

### 1.4 并发安全问题

Webhook Server 同时面对 WebApp 写入、Duo 板轮询和 ACK 确认。即使当前只有一个机器人端，也可能出现并发请求，例如：

- `/emotion` 正在写入新动作时，`/poll/{client_id}` 同时读取队列
- Duo 板刚轮询到命令，还没 ACK，新一轮情绪输入又到达
- 多个客户端同时访问自己的队列

如果队列只是普通 list，没有保护机制，就可能出现数据竞争：重复取到同一条命令、ACK 找不到命令、队列状态被覆盖等问题。

解决方式是在队列管理层使用 `threading.Lock` 包住关键操作：

- `add_action()`：写入新命令时加锁，保证队列 append 和旧命令清理不会被打断
- `poll_action()`：查找 `pending` 命令并标记为 `processing` 时加锁，保证同一条命令不会被多个请求重复取走
- `acknowledge_action()`：根据 `command_id` 标记 completed 时加锁，避免状态更新与清理操作冲突
- `cleanup_old()`：清理过期命令时加锁，避免和读写队列同时发生

这部分的关键不是代码复杂度，而是把队列状态变更视为一个原子操作。只要"读取并修改状态"不是原子的，轮询架构就会出现偶发错误。

---

### 1.5 实时性测试数据

这一阶段对完整链路做了基础延迟估算：

| 环节 | 耗时 |
|------|------|
| 情绪分析 / 动作生成 | 约 2 秒 |
| Duo 板轮询等待 | 0-2 秒 |
| 总延迟 | 2-4 秒，平均约 3 秒 |

其中 AI 调用仍然是主要耗时来源，轮询等待是架构选择带来的固定延迟。当前轮询间隔为 2 秒，如果需要更快响应，可以把轮询间隔降到 1 秒，总延迟可以压缩到约 2-3 秒，但会增加机器人端和服务器的请求频率。

从当前项目阶段看，2 秒轮询是比较平衡的选择：延迟可接受，服务端压力较低，Duo 板实现也简单。

### 1.6 技术选型

**为什么继续使用 OpenClaw Skill：**

- Skill 能承载机器人行为生成能力，减少自研 Agent 调度逻辑
- 能把重点放到故事解析、动作序列和行为表达，而不是会话运行环境
- 与本地 Agent 平台生态更容易衔接

**为什么使用 qwen3.5-omni-plus-realtime：**

- WebSocket 实时连接，适合低延迟语义理解
- 中文故事和情绪表达理解能力较强
- 输出结构化 JSON 后可以直接进入动作队列

## 2. AI 生成代码实战

### 2.1 提示词设计

在这一阶段，AI 辅助的重点不是再生成情绪映射表，而是围绕 Webhook Server 的工程约束生成代码。提示词中明确了端点、队列、状态和错误处理边界：

```
请实现一个 Python HTTP 服务，用于接收机器人动作指令并维护动作队列。

要求：
1. POST /emotion：接收文本，调用 robot-behavior 生成 action_sequence
2. GET /poll/{client_id}：客户端轮询 pending 命令
3. POST /ack/{client_id}：客户端确认 command_id 已执行
4. 每个 client_id 独立队列
5. 每条命令包含 command_id、action_sequence、emotion、created_at、status
6. 使用 threading.Lock 保证队列读写安全
7. 返回 JSON，不输出多余文本
```

相比一句"帮我写 Webhook 服务"，这种提示词直接把接口、数据结构和并发约束写清楚，生成结果更接近可落地代码。

### 2.2 VIBE 框架的实际迭代过程

这里不展开 VIBE 的定义，只记录它在本项目中的实际迭代过程。

**第 1 轮：跑通基础链路**

- 目标：实现 `/emotion`、`/poll/{client_id}`、`/ack/{client_id}` 三个核心端点
- 结果：WebApp 可以写入动作，Duo 板可以轮询到动作
- 发现问题：错误处理不足，异常情况下返回格式不稳定

**第 2 轮：补齐错误处理和状态字段**

- 修改：为非法 JSON、缺少字段、情绪分析失败增加明确错误返回
- 修改：队列项增加 `status`，从只存动作序列改为存完整命令对象
- 发现问题：轮询和 ACK 之间缺少明确追踪字段，调试时难以定位是哪条命令出错

**第 3 轮：引入 command_id、日志和超时策略**

- 修改：每条命令生成唯一 `command_id`
- 修改：日志记录入队、轮询、ACK、清理等关键动作
- 修改：情绪分析调用设置超时，避免外部 API 卡住整个请求
- 结果：链路具备了基本可观测性，可以定位命令从生成到执行确认的完整过程

这三轮迭代的价值在于：从"能跑"逐步变成"能调试、能定位、能稳定联调"。

---

### 2.3 SPEC 框架的实际迭代过程

队列管理不是一次设计完成的，而是从最小版本逐步演进。

**V1：单队列原型**

```python
class ActionQueue:
    def __init__(self):
        self.queue = []

    def add(self, action):
        self.queue.append(action)
```

V1 的作用是验证动作可以被写入和取出，但问题很明显：

- 不支持多个客户端
- 没有命令 ID，无法追踪
- 没有状态管理，无法区分待执行和已完成
- 没有锁，并发访问不安全

**V2：多客户端 + command_id**

- 队列结构从单 list 改为按 `client_id` 分组
- 每条命令生成 `command_id`
- `/poll/{client_id}` 只读取对应客户端队列

V2 解决了多客户端和命令追踪问题，但仍然缺少状态生命周期。

**V3：状态管理 + 线程安全**

- 队列项增加 `pending`、`processing`、`completed` 状态
- `poll_action()` 返回命令时将其标记为 `processing`
- `acknowledge_action()` 根据 `command_id` 标记完成
- 所有队列读写操作使用 `threading.Lock`

V3 才是真正适合联调的版本：它能支持写入、轮询、执行、确认和清理的完整命令生命周期。

## 3. 技术收获

### 3.1 WebSocket 实时调用

本阶段仍然使用 qwen3.5-omni-plus-realtime 做语义理解。调用流程是：

```
建立连接 → 配置会话 → 发送消息 → 接收响应 → 关闭
```

WebSocket 相比普通 HTTP 请求少了一些重复连接开销，适合频繁调用的语义分析场景。但对整体延迟影响最大的仍然是模型响应时间，因此 Webhook 队列需要做好超时、降级和日志记录。

---

### 3.2 轮询架构的权衡

轮询和推送的差异在这个项目里非常直接：

| 方案 | 优点 | 成本 |
|------|------|------|
| 推送 | 实时性更好 | 需要处理公网访问、连接保持、设备在线状态 |
| 轮询 | 部署简单，穿透内网更容易 | 增加 0-2 秒等待时间 |

本阶段选择轮询，是因为机器人端联调优先级高于极限实时性。只要轮询间隔、队列状态和 ACK 机制设计清楚，这个方案可以稳定支持动作执行。

---

### 3.3 工具与实现点

- OpenClaw：承载 Skill 能力和 Agent 运行上下文
- qwen3.5-omni-plus：负责语义理解和结构化输出
- Webhook Receiver：维护动作队列和机器人端轮询接口
- `threading.Lock`：保证队列读写和状态变更的并发安全
- `command_id`：串起命令生成、轮询、执行和确认过程

## 4. 阶段小结与遗留问题

这一阶段完成了 Webhook Receiver 和动作队列的工程落地，解决了 Skill 输出动作序列后如何可靠传递给机器人端执行的问题。通过 `/emotion`、`/poll/{client_id}` 和 `/ack/{client_id}`，系统形成了"生成动作 → 入队 → 轮询 → 执行 → 确认"的完整闭环。

当前遗留的核心问题不在队列传输，而在动作生成本身：系统对复杂故事的处理仍然停留在单一情绪标签层面。对于短文本，情绪标签可以有效驱动动作；但对于包含多个情节节点的故事，单一标签会丢失叙事时序结构，导致动作序列缺少"先发生什么、再发生什么"的表达能力。

下一阶段将通过两阶段故事解析方案解决这个问题：先按情节节点拆分故事，再逐段生成动作序列，最后合并为完整动作表达。
