## 一、核心功能实现与技术选型

### 1.1 系统架构总览

我正在做的这个项目是一个**云边协同的情绪感知陪伴机器人系统**。整体架构如下：

```
┌─────────────────┐                      ┌─────────────────┐
│   边端 Duo S     │                      │   云端 Agent     │
│  - 音频采集      │  ───── WebSocket ──► │  - 情绪识别     │
│  - 帧发送        │                      │  - 对话生成     │
│  - 舵机执行      │  ◄──── JSON 响应 ──  │  - 动作决策     │
└─────────────────┘                      └─────────────────┘
```

**核心设计原则**：

- **云边分离**：云端只负责"思考"（决策），边端负责"执行"（硬件控制）
- **动作抽象**：云端输出动作标签（如 `happy`、`wave`），而非舵机角度
- **JSON 强约束**：所有模型输出必须是纯 JSON，无解释、无多余文本

---

### 1.2 核心模块实现

#### 模块 1：请求/响应数据模型（`app/models/`）

使用 Pydantic 定义严格的协议约束：

**为什么用 Pydantic**：

- 自动类型校验，防止非法数据流入
- 自动生成 OpenAPI 文档
- `Literal` 类型约束 action 取值范围

---

#### 模块 2：ChatPipeline 流程编排（`app/pipeline/chat_pipeline.py`）

这是整个系统的"中枢神经"，负责串联所有模块：

**管道模式的优势**：

- 每个模块职责单一，便于独立测试
- 可以轻松替换某个模块（例如切换 LLM 供应商）
- 性能诊断时可以精准定位耗时环节

---

#### 模块 3：LLM 客户端封装（`app/service/llm/client.py`）

#### **关键设计点**：

- 使用 `asyncio.wait_for` 实现超时控制
- 封装 DashScope API（兼容 OpenAI 接口）
- 返回原始 content，由上层解析 JSON

---

#### 模块 4：决策引擎（`app/service/decision/decision_engine.py`）**为什么用规则而非 LLM**：

- 降低不确定性（LLM 可能输出非法 action）
- 响应速度快（< 5ms）
- 可解释性强，便于调试

---

#### 模块 5：JSON 协议构建与校验（`app/service/json_protocol/response_builder.py`）

这是云端输出前的**最后一道防线**

**为什么需要这层校验**：

- LLM 可能输出非法 action（例如直接输出舵机角度）
- 边端对协议极其敏感，非法 JSON 会导致执行错误
- 兜底策略确保永远返回合法响应

## 三、我对云边协同架构的技术理解

### 3.1 为什么选择云边协同？

在项目中，我深刻体会到**云边协同**的核心价值：

| 维度     | 纯云端方案       | 纯边端方案     | 云边协同         |
| -------- | ---------------- | -------------- | ---------------- |
| 响应延迟 | 高（需上传音频） | 低             | 中               |
| 计算能力 | 强（可跑大模型） | 弱（资源受限） | 云端负责复杂计算 |
| 硬件依赖 | 无               | 强             | 边端专注执行     |
| 可扩展性 | 强               | 弱             | 云端可服务多设备 |

**我的项目定位**：云端负责"思考"（情绪识别、对话生成、动作决策），边端负责"执行"（音频采集、舵机控制、TTS 播放）。

---

### 3.2 动作抽象：云端不碰硬件

这是我认为最关键的设计决策。

**错误做法**（云端输出舵机角度）：

```json
{"action": "servo_angle: 45"}  // ❌ 云端耦合硬件细节
```

**正确做法**（云端输出动作标签）：

```json
{"action": "wave"}  // ✅ 云端只负责语义决策
```

**好处**：

1. 云端代码与硬件解耦，可服务不同型号的边端设备
2. 边端可以独立升级动作库（例如优化 wave 的舵机轨迹）
3. 同一套云端代码可以适配舵机、屏幕、LED 等不同执行器

---

### 3.3 JSON 强约束：为什么这么重要？

在云边通信中，**JSON 协议的稳定性**直接决定系统可靠性。

我踩过一个坑：早期 LLM 偶尔会输出这样的内容：

```json
{
  "emotion": "happy",
  "reply_text": "你好呀！"
}
// 以上是 JSON，下面是额外解释
这个回复表达了积极的情绪，适合用于问候场景。
```

边端解析时直接报错，因为收到了 JSON 之外的内容。

**解决方案**：

1. 在 Prompt 中强调"Output STRICT JSON only (no markdown, no extra text)"
2. 用正则提取第一个 `{...}` 块（`parse_strict_json` 函数）
3. 用 Pydantic 校验最终输出（`build_chat_response` 函数）

---

### 3.4 管道模式：让系统可维护、可测试

我在这个项目中采用**管道模式**（Pipeline Pattern）编排请求处理：

```
用户请求 → ASR 模块 → LLM 模块 → 决策引擎 → JSON 协议构建 → 响应
```

**好处**：

1. **可测试**：每个模块可以独立写单元测试
2. **可替换**：例如切换 LLM 供应商只需改 `LLMClient`
3. **可观测**：每个模块记录耗时，便于性能诊断
4. **可降级**：LLM 超时时可以跳过情绪识别，返回默认回复

---

### 3.5 关于性能的思考

在项目中，我发现**LLM 调用耗时**是主要瓶颈（800ms ~ 3500ms）。

**优化方向**：

| 方案                              | 预期收益          | 成本                   |
| --------------------------------- | ----------------- | ---------------------- |
| 切换到更快的模型（qwen3.5-flash） | -30%~50% 响应时间 | 情绪识别准确率可能略降 |
| 限制历史对话轮数（MAX_HISTORY=5） | -10%~20% 耗时     | 上下文理解能力略降     |
| 高频问题走本地缓存                | 命中时 < 50ms     | 缓存覆盖率有限         |
| 流式输出                          | 改善用户体验      | 总响应时间不变         |

**我的选择**：优先切换模型，中期添加缓存，长期优化 Prompt 工程。

## 四、架构演进：从三次调用到一次调用

这是我在项目中最大的架构优化，也是和 AI 助手多轮讨论后得出的方案。

### 4.1 初始架构：三次 LLM 调用

项目最开始，我设计的流程是**串行三次 LLM 调用**：

```
用户输入 
  │
  ▼
┌─────────────────┐
│ 1. 情绪识别 LLM  │ → {"emotion": "happy"}
└─────────────────┘
  │
  ▼
┌─────────────────┐
│ 2. 对话生成 LLM  │ → {"reply_text": "你好呀！"}
└─────────────────┘
  │
  ▼
┌─────────────────┐
│ 3. TTS 文本优化   │ → {"tts_text": "你好呀～"}
└─────────────────┘
  │
  ▼
决策引擎 → 返回响应
```

**问题非常明显**：

- 三次 API 调用，延迟叠加（3 × 800ms = 2400ms+）
- 三次 token 消耗，成本翻倍
- 三次独立调用，情绪可能不一致（第一次识别为 happy，第二次对话生成时却按 neutral 处理）

---

### 4.2 架构优化：合并为一次调用

```python

2. Classify the user's emotion into one of: ["happy", "sad", "neutral"]
3. Write a friendly, supportive reply that matches the detected emotion.
4. Keep replies short and natural for spoken TTS.

Output STRICT JSON only (no markdown, no extra text):
{{
  "emotion": "one of the allowed labels",
  "reply_text": "...",
  "tts_text": "..."
}}

User input: {user_input}
"""
```

**优化后的流程**：

```
用户输入 
  │
  ▼
┌─────────────────────────────┐
│ 单次 LLM 调用（统一 Prompt）  │
│ → {"emotion": "happy",      │
│     "reply_text": "...",    │
│     "tts_text": "..."}      │
└─────────────────────────────┘
  │
  ▼
决策引擎 → 返回响应
```

| ------------ | ---------- | -------- | -------- |

| 平均响应时间 | ~2400ms    | ~900ms   | -62%     |
| Token 消耗   | 3×         | 1×       | -67%     |

### 4.4 情感表达三重奏系统

在单次调用的基础上，我们进一步提出了**情感三重奏（Emotion Trio）**的概念：

```
┌─────────────────────────────────────┐
│       情感表达三重奏                │
├─────────────────────────────────────┤
│  1. 语言回复（reply_text）          │  ──→ 文字内容表达情感
│  2. TTS 语调（tts_text）            │  ──→ 语音语调传递情感
│  3. 机器人动作（action）            │  ──→ 肢体动作强化情感
└─────────────────────────────────────┘
```

**三者协同工作**：

- **语言**：LLM 根据情绪生成匹配的回复内容（如 happy 时用积极的措辞）
- **TTS**：`tts_text` 可以添加语气词、标点符号来调整语音合成效果
- **动作**：决策引擎将 `emotion` 映射为 `action`（happy→happy 动作）

**示例**（用户说"我今天升职了！"）：

```json
{
  "emotion": "happy",
  "reply_text": "太棒了！恭喜你！",
  "tts_text": "哇～太棒了！恭喜你！",  // 添加语气词
  "action": "happy"  // 机器人做开心动作
}
```

---

### 4.5 决策引擎：确定性的规则设计

在三重奏中，**决策引擎**负责将情绪映射为动作：**为什么用规则而非 LLM**：

- **确定性**：相同输入永远得到相同输出
- **响应速度**：< 5ms（vs LLM 的 800ms+）
- **可解释性**：映射关系清晰，便于调试
- **成本**：零 token 消耗

**未来升级方向**：可以引入"情绪强度"概念，例如：

- `happy` + 强度 0.8 → `happy` 动作（大幅度）
- `happy` + 强度 0.3 → `nod` 动作（轻微点头）

### 演进方向

项目从自研 Agent 迁移到 OpenClaw Skill 的核心决策逻辑，是将开发重心从底层 Agent 通信、状态管理和调度服务，转向更贴近机器人体验本身的能力建设。

1. **通信与调度平台化**：减少对自研 WebSocket 服务、Agent 状态管理和请求调度逻辑的维护投入。
2. **行为生成专业化**：把更多精力投入到故事解析、情绪识别、动作序列生成和动作编排质量上。
3. **机器人表达精细化**：围绕动作时长、动作衔接、叙事情绪变化和硬件执行效果，持续优化机器人行为表达。
4. **生态集成能力增强**：借助 OpenClaw Skill 的结构化能力，将机器人行为模块更自然地接入 Agent 平台生态。
