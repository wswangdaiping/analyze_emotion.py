### 优化 1：修复分段阈值问题 ⭐（完成-2026-05-15 ykx）

**问题**：超过 4 句话的连续叙事故事，`segment_story()` 倾向于不分段，导致 `last_action` 跨段传递机制无法生效，分段架构优势丢失。

**方案**：修改 `SEGMENT_SYSTEM_PROMPT`，增加分段规则：
- 超过 3 句话必须至少分 2 段
- 场景变化 / 动作性质变化 / 情绪强度变化 / 时间推进词 → 触发新段
- 每段 1-3 句话
- 单句或双句仍保持单段

**文件**：`skills/robot-behavior/scripts/analyze_emotion.py`（`SEGMENT_SYSTEM_PROMPT`）

**工作量**: 小 | **优先级**: 最高（已有明确方案，直接改 prompt）

### 优化 2：情绪弧线感知（完成-2026-05-15 ykx）

**问题**：多段故事只有一个综合情绪标签，无法体现情绪演变过程。

**方案**：每段单独标注情绪，最终返回情绪变化路径（如 `neutral → nervous → happy`），而非单一标签。`detect_emotion()` 已经在逐段检测，需要的是在最终输出中保留路径。

**文件**：`analyze_emotion.py`（`build_story_actions()` 的 `emotion_detected` 输出逻辑）

**工作量**: 小 | **优先级**: 高

#### 提示词：
项目背景：

请按以下设计修改 analyze_emotion.py：

【方向十：AI 调用降级链】

修改 call_dashscope_json() 的异常处理，实现以下降级链：

第一级：正常调用 dashscope API
第二级：超时或网络错误 → 自动重试一次（等待 2 秒后重试）
第三级：重试仍失败 → 返回默认动作 [0]

每次降级时在 stderr 打印降级原因，格式：
[FALLBACK] 原因 → 降级到第X级

在最终返回的 JSON 里新增字段：
"fallback_level": 0  # 0=正常, 1=重试成功, 2=默认降级

【格式要求】
- 保持所有现有接口不变
- fallback_level 字段仅内部输出，server.py 对外响应时剥离
  （和 segments 字段一样的处理方式）
- 只修改 analyze_emotion.py

#### codex 实现思路：
我准备这样做，范围控制在“方向十”本身，不碰别的优化项。

  实现思路

  核心改 /D:/Project_SE/openclaw-project-export/skills/robot-behavior/scripts/analyze_emotion.py:202 里的 call_dashscope_json()：

  1. 把当前 DashScope WebSocket 调用封装成一次 attempt。
  2. 第一次正常调用，成功则 fallback_level=0。
  3. 如果遇到超时、网络错误、连接错误这类异常：
      - stderr 打印：[FALLBACK] 网络/超时原因 → 降级到第1级
      - time.sleep(2)
      - 自动重试一次。
  4. 重试成功：
      - 返回正常 data
      - 标记 fallback_level=1
  5. 重试仍失败：
      - stderr 打印：[FALLBACK] 重试失败原因 → 降级到第2级
      - 返回失败结果并标记 fallback_level=2
      - 由现有调用方继续走已有兜底：generate_actions() 返回 [0]，segment_story() 回退原文，detect_emotion() 用规则情绪。

  这里我不会简单让 call_dashscope_json() 对所有场景都直接返回 [0]，因为它不只服务动作生成，也服务分段和情绪识别。直接返回 [0] 会破坏分段/情绪调用的数据类型。更稳的做法是：动作生成失败时默认 [0]，分段和
  情绪识别走各自已有兜底。

  fallback_level 怎么放进最终 JSON

  我会加一个很小的内部状态，例如本次分析过程的最高降级级别：

  - 全程正常：fallback_level: 0
  - 任意一次 API 调用第一次失败但重试成功：fallback_level: 1
  - 任意一次重试后仍失败并进入默认/规则兜底：fallback_level: 2

  最终在 build_story_actions() 返回值里新增：

  "fallback_level": 0

  server.py 是否要改

  方向十写了“只修改 analyze_emotion.py”，同时要求 server.py 对外剥离。当前 /D:/Project_SE/openclaw-project-export/services/webhook-receiver/server.py:367 本来就是白名单式返回：只取 emotion、
  action_sequence、command_id、client_id，不会把 segments、emotion_arc、未来的 fallback_level 原样透出。所以我预计不需要改 server.py。

     预期：fallback_level 为 0。

  3. DashScope 异常路径验证

     python skills\robot-behavior\scripts\analyze_emotion.py --provider dashscope --api-key bad_key --timeout 1 --pretty --input "小明开心地挥手。"

     预期：
      - stderr 有 [FALLBACK] ... → 降级到第1级
      - 重试失败后有 [FALLBACK] ... → 降级到第2级
      - stdout 仍是合法 JSON
      - 最终动作能兜底到 [0] 或保持现有规则兜底
      - fallback_level 为 2

## Brooks-Lint 代码健康检查问题整理

本节根据 Brooks-Lint Health Dashboard 的检查结果，对项目当前可作为后续工程化优化方向的问题进行整理。以下内容主要用于项目文档、答辩补充材料和后续迭代规划，不表示当前功能不可用，而是指出在测试可靠性、代码可维护性和环境迁移方面可以继续增强的点。

### 问题 1：批量回归测试未形成有效失败门禁

- 问题类型：测试有效性 / 回归门禁不足
- 涉及文件：`tests/batch_story_test.py`
- 具体表现：批量故事测试脚本会记录每个用例的期望动作序列和实际动作序列，但当脚本调用失败、结果不一致或部分用例异常时，主要表现为继续生成报告，而不是明确以非零退出状态结束。
- 可能影响：测试看起来已经执行并产出报告，但实际没有强制验证核心结果是否正确，可能导致动作序列、情绪识别或长故事分段出现回归时未能及时暴露。
- 优化方向：将该脚本从“报告生成工具”进一步改造成“回归测试门禁”，对关键字段、退出状态、动作序列尾部 `0`、重点用例的期望与实际结果增加断言；失败用例应累计并最终返回非零退出码。
- 优先级：第一优先

### 问题 2：情绪分析核心流程函数较长，职责混合

- 问题类型：可维护性 / 核心流程复杂度偏高
- 涉及文件：`skills/robot-behavior/scripts/analyze_emotion.py`
- 具体表现：核心流程中同时包含 DashScope 调用、WebSocket 协议交互、JSON 解析、fallback 降级、故事分段、情绪识别和动作编排等逻辑，部分函数承担的职责较多。
- 可能影响：后续修改模型调用方式、提示词策略、异常处理或动作生成规则时，容易牵动同一段核心代码，增加理解成本和回归风险。
- 优化方向：逐步拆分模型调用、JSON 响应解析、故事分段、动作编排等职责，形成更清晰的内部模块边界；拆分前建议先补充特征化测试，锁定当前已有行为。
- 优先级：第二优先

### 问题 3：WebhookHandler 职责偏重

- 问题类型：服务端结构 / 单一职责不足
- 涉及文件：`services/webhook-receiver/server.py`
- 具体表现：`WebhookHandler` 同时承担路由分发、鉴权、JSON 请求解析、情绪分析脚本调用、动作队列写入、轮询确认和响应格式封装等职责。
- 可能影响：当 API 字段、鉴权策略、队列策略或情绪分析调用方式发生变化时，多个原因都会集中修改同一个类，容易扩大改动范围，也不利于单独测试服务端业务逻辑。
- 优化方向：保留 HTTP Handler 作为薄适配层，将请求解析、业务调度、动作队列管理和响应封装逐步拆分为独立函数或服务类，使后续接口变更更集中、更容易验证。
- 优先级：第三优先

### 问题 4：部分脚本存在部署路径或运行路径硬编码

- 问题类型：环境迁移 / 路径管理不统一
- 涉及文件：`skills/robot-behavior/scripts/director_video.py`、`tests/batch_story_test.py`
- 具体表现：部分脚本中存在面向服务器部署目录的固定路径，例如直接引用 `/home/admin/.openclaw/workspace/...`，而当前项目也会在本地 Windows 导出目录中维护和测试。
- 可能影响：同一脚本在本地、服务器、Windows、Unix 等环境之间迁移时需要手动修改路径，容易出现“某台机器能跑，换环境失败”的情况。
- 优化方向：统一路径管理方式，优先通过 repo root 推导、环境变量或命令行参数传入关键路径；将部署目录留在部署配置或说明文档中，不直接固化到通用脚本逻辑里。
- 优先级：第四优先

### 问题 5：缓存文件污染工作区

- 问题类型：仓库卫生 / 生成文件管理
- 涉及文件：`.gitignore`、`services/webhook-receiver/__pycache__/`、`skills/robot-behavior/scripts/__pycache__/`
- 具体表现：工作区中出现 `__pycache__`、`.pyc` 等 Python 缓存文件目录，且当前忽略规则未覆盖这类生成文件。
- 可能影响：版本管理状态中会混入与功能无关的缓存目录，影响后续代码审查、提交判断和验证过程；在个别环境中，缓存目录权限还可能干扰语法检查或编译缓存写入。
- 优化方向：在 `.gitignore` 中补充 `__pycache__/`、`*.pyc` 等规则，并清理已经生成的缓存目录，保持工作区只关注源代码、文档和必要配置。
- 优先级：第五优先

### 建议优化顺序

1. 第一优先：修复批量回归测试，使失败用例真正导致测试失败，提升后续修改的安全性。
2. 第二优先：拆分 `analyze_emotion.py`，将模型调用、JSON 解析、故事分段、动作编排等逻辑逐步解耦。
3. 第三优先：拆分 `WebhookHandler`，将请求解析、业务调用、队列管理和响应封装分离。
4. 第四优先：统一路径管理，使用 repo root、环境变量或命令行参数替代硬编码部署路径。
5. 第五优先：完善 `.gitignore`，清理 `__pycache__` 和 `*.pyc` 等缓存文件。
