所以优化方向应该是**动作密度和节奏控制**，而不是更复杂的算法。

具体来说有三个方向，你选你觉得最有价值的：

**方向一：动作数量根据故事长度动态控制** 短故事（1-2句）→ 2-3个动作 中等故事（3-5句）→ 3-5个动作 长故事（5句以上）→ 每段2个动作，控制总数不超过8个

**方向二：引入"节奏标签"** 在prompt里让LLM给每段故事标注节奏：快节奏/慢节奏/高潮 快节奏段落动作数多，慢节奏段落动作数少，高潮段落用强动作

**方向三：动作时长加权** 给每个动作标注预估执行时长，生成序列时控制总时长在合理范围内（比如30秒以内）

## 架构与集成

**方向四：OpenClaw Agent 深度集成** 当前 server.py 通过 subprocess 硬调用 analyze_emotion.py，完全绕过了 OpenClaw 的 Skill 调度系统。改为由 OpenClaw Agent 根据 SKILL.md 自动匹配并链式调用 robot-behavior → json-webhook-skill，收益：多轮会话上下文、自动错误恢复、skill 可组合。已通过 `?mode=agent` 完成小步验证入口，待 OpenClaw 侧配置落地。

**方向五：多渠道情绪输入** 利用 OpenClaw channel plugins（Discord/Telegram/飞书/QQ/企微），让用户在这些平台上直接发送情绪文本触发机器人动作，不需要 WebApp 作为唯一入口。适合演示场景和远程交互。

**方向六：会话级情绪追踪** 利用 OpenClaw session 管理，为同一用户的连续情绪输入维护上下文。例如：用户先发"我很难过"，机器人做 sad 动作；用户再发"现在好多了"，系统能结合上下文判断这是 sad→happy 的过渡，而非孤立分析。当前实现每次都是独立分析，缺少连续性。

## 性能与成本

**方向七：本地规则优先 + AI 兜底** 大多数情绪输入是简单的关键词匹配（"我很开心"、"我很难过"），不需要每次都调 DashScope API。改为：mock/规则引擎先匹配 → 命中直接返回动作（0 延迟、0 成本）→ 未命中或置信度低时再走 AI。在 `analyze_emotion.py` 中已有 `--provider mock` 基础，可扩展为混合模式。

**方向八：轮询优化（长轮询/SSE）** 当前 Milk DuoS 每 2 秒短轮询一次，大部分请求返回 `no_action`，浪费带宽和电量。改为 HTTP 长轮询（hold 请求直到有新动作或超时），在不改变 DuoS 硬件代码的前提下减少 90% 的空请求。

**方向九：动作队列持久化** 当前队列是内存字典，服务重启后所有待执行命令丢失。改为 SQLite 文件或 `.runtime/action-queue/` 下的 JSON 文件持久化，崩溃恢复后能继续投喂未完成的动作。

## 稳定性

**方向十：AI 调用降级链** 当前 DashScope API 超时或失败时直接返回 500。应实现降级链：AI 超时(15s) → 重试 1 次 → 失败则降级到本地规则引擎(mock) → 规则也失败则返回默认 [0]。每层降级都记录日志，避免静默丢失。

**方向十一：队列健康监控** 新增 `GET /queue/stats` 端点，暴露每个客户端的队列深度、pending/processing/completed 数量、最老命令年龄。便于排查"机器人不动了"是没收到情绪还是队列阻塞。

## 功能扩展

**方向十二：动作反馈闭环** DuoS 端 ACK 目前只确认"执行了"，不反馈执行质量。扩展 ACK 协议让硬件上报每个动作的实际耗时、是否被中断，用于后续优化动作时长加权（方向三）的参数。
